# Class 4: Security

> **User story** · As the product owner, I want the platform open for browsing but locked for editing, so that going public never puts the data at risk.

Our API is wide open. Since class 3, anyone who can reach `/graphql` can delete movies, rewrite plots, and force-delete people, no questions asked. Before this service could ever face real users, it needs two things: **authentication** (who is calling?) and **authorization** (what is this caller allowed to do?). In this class we add both, the way a production Spring application does: Spring Security for the machinery, JWT tokens for stateless API authentication, and role-based rules for the sensitive operations.

By the end of this class, you will:

- Understand Spring Security's secure-by-default behavior and why "everything is blocked" is the right starting point
- Configure a `SecurityFilterChain` that permits exactly what the API needs
- Authenticate users with a `login` mutation that issues JWT tokens
- Protect mutations with role-based authorization

## 1. Add the dependency: secure by default

We begin with the dependency alone, without any configuration:

`pom.xml`

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>
```

Restart the application and verify the effect:

- **GraphiQL** is no longer reachable: the browser is redirected to a login page generated by Spring Security.
- **The frontend** cannot reach the API: every request to `/graphql` is rejected with `401 Unauthorized`.
- The **H2 console** is locked as well.
- The application log contains a new line:

```
Using generated security password: f53430aa-9c70-4bce-9a5f-...
```

This is Spring Security's **secure-by-default** behavior. As soon as the starter is on the classpath, an auto-configured filter chain protects every endpoint, creates a single in-memory user (`user`, with the generated password from the log), enables CSRF protection, and serves a generated login form. Nothing is exposed by accident; whatever should be reachable must be opened deliberately.

None of these defaults fit a GraphQL API: we do not want form login, server-side sessions, or a single hardcoded user. The *posture*, however, is correct. Blocked is the right starting state; in the following steps we allow precisely what we intend, and nothing more.

## 2. The security configuration

The starter already contains a small configuration class, `SecurityConfig`, whose only job so far was to provide the `PasswordEncoder` bean that `DataInitializer` and `UserService` use for password hashing. It now takes on its central role: we define the filter chain in it. Providing our own `SecurityFilterChain` bean replaces Spring's auto-configured one, which is how we take control of the rules:

`src/main/java/com/graphqlguy/moviedb/config/SecurityConfig.java`

```java
package com.graphqlguy.moviedb.config;

import com.graphqlguy.moviedb.security.JwtAuthFilter;
import lombok.RequiredArgsConstructor;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.http.SessionCreationPolicy;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import org.springframework.security.crypto.password.PasswordEncoder;
import org.springframework.security.web.SecurityFilterChain;
import org.springframework.security.web.authentication.UsernamePasswordAuthenticationFilter;

@Configuration
@RequiredArgsConstructor
public class SecurityConfig {

    private final JwtAuthFilter jwtAuthFilter;

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
                .csrf(csrf -> csrf.disable())
                .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
                .authorizeHttpRequests(auth -> auth.anyRequest().permitAll())
                .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class);
        return http.build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

Each of the three settings is a deliberate decision, and each deserves justification:

- **CSRF is disabled.** CSRF protection guards against attacks where a malicious website submits a request to your server using the victim's session cookie. Our API authenticates with JWT tokens sent in a header, not with cookies, and holds no sessions, so this class of attack does not apply.
- **Session management is `STATELESS`.** A JWT carries all authentication information inside itself, so the server does not need to remember anything between requests. This is simpler, and it scales: any server in a cluster can validate a token without consulting a shared session store.
- **All requests are permitted at the HTTP level.** This may appear to defeat the purpose. The reason is architectural: GraphQL has a single endpoint, and unauthenticated users must be able to reach it for public queries such as `movies`. URL-based rules therefore cannot express our requirements. Authorization will instead be enforced at the **method level**, on the specific operations that need protection.

The `addFilterBefore` line registers a filter we have not written yet, which is the next step. (You will also notice the generated-password line still appears in the log; it belongs to Spring's default in-memory user, which our configuration never uses.)

## 3. The JWT authentication filter

A JWT (JSON Web Token) is a cryptographically signed token that carries identity information. The flow: when a user logs in, the server creates a token containing the username and role, signs it with a secret key, and returns it. On every subsequent request, the client sends the token in the `Authorization` header, and the server verifies the signature, without a database lookup and without any session.

The starter already contains `JwtUtil` (in the `security` package), which generates and validates these tokens using the secret from `application.yaml`. What we write is the filter that applies it to incoming requests:

`src/main/java/com/graphqlguy/moviedb/security/JwtAuthFilter.java`

```java
package com.graphqlguy.moviedb.security;

import jakarta.servlet.FilterChain;
import jakarta.servlet.ServletException;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import lombok.RequiredArgsConstructor;
import org.springframework.security.authentication.UsernamePasswordAuthenticationToken;
import org.springframework.security.core.authority.SimpleGrantedAuthority;
import org.springframework.security.core.context.SecurityContextHolder;
import org.springframework.stereotype.Component;
import org.springframework.web.filter.OncePerRequestFilter;

import java.io.IOException;
import java.util.List;

@Component
@RequiredArgsConstructor
public class JwtAuthFilter extends OncePerRequestFilter {

    private final JwtUtil jwtUtil;

    @Override
    protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response,
                                    FilterChain filterChain) throws ServletException, IOException {
        String authHeader = request.getHeader("Authorization");

        if (authHeader != null && authHeader.startsWith("Bearer ")) {
            String token = authHeader.substring(7);
            if (jwtUtil.isTokenValid(token)) {
                String username = jwtUtil.extractUsername(token);
                String role = jwtUtil.extractRole(token);
                var auth = new UsernamePasswordAuthenticationToken(
                        username, null,
                        List.of(new SimpleGrantedAuthority("ROLE_" + role))
                );
                SecurityContextHolder.getContext().setAuthentication(auth);
            }
        }

        filterChain.doFilter(request, response);
    }
}
```

Points worth noting:

- `OncePerRequestFilter` guarantees the filter runs exactly once per request, regardless of internal forwards.
- If a valid token is present, the filter populates the `SecurityContextHolder`: from this moment on, the rest of the application knows who the caller is and which role they hold. The `ROLE_` prefix is a Spring Security convention that `hasRole(...)` checks rely on.
- If the token is absent or invalid, the filter does **not** reject the request. It simply leaves the context empty and lets the request continue anonymously. Whether an anonymous caller is acceptable is an authorization decision, made later and per operation, not the filter's concern.

## 4. Enable method security

One last switch. Because our authorization will live on methods rather than URLs, Spring must be told to honor those annotations:

`src/main/java/com/graphqlguy/moviedb/MovieDbApplication.java`

```java
import org.springframework.security.config.annotation.method.configuration.EnableMethodSecurity;

@EnableMethodSecurity
@SpringBootApplication
@ConfigurationPropertiesScan
public class MovieDbApplication {
```

Without `@EnableMethodSecurity`, every `@PreAuthorize` annotation in the codebase is **silently ignored**: no error, no warning, no protection. This is one of the most common Spring Security mistakes, and it is worth remembering that the symptom is an application that appears to work perfectly.

Restart the application. GraphiQL, the frontend, and the H2 console are all reachable again, but the situation is fundamentally different from where the class started: every request now passes through the JWT filter and can carry an authenticated identity. What is still missing is a way for a user to obtain a token in the first place.

## 5. Obtaining a token: the login mutation

Authentication enters the schema as a mutation. It returns the token together with the authenticated user, so a client can store one and display the other after a single request:

`src/main/resources/graphql/schema.graphqls`

```graphql
type Mutation {
    # ...existing fields...
    login(input: LoginInput!): AuthResponse!
}

input LoginInput {
    username: String!
    password: String!
}

type AuthResponse {
    token: String!
    user: User!
}

type User {
    id: ID!
    username: String!
    email: String
    role: Role!
}

enum Role {
    USER,
    ADMIN
}
```

The service side is starter material and worth reading before you continue: `UserService.login` looks the user up, verifies the password with the `PasswordEncoder` (the seeded passwords are BCrypt hashes, never plain text), and calls `JwtUtil.generateToken` to produce the signed token. Note that a wrong username and a wrong password produce the same error message; revealing which of the two was wrong would help an attacker enumerate accounts.

The controller is a new class and, by now, entirely familiar:

`src/main/java/com/graphqlguy/moviedb/user/UserController.java`

```java
package com.graphqlguy.moviedb.user;

import lombok.RequiredArgsConstructor;
import org.springframework.graphql.data.method.annotation.Argument;
import org.springframework.graphql.data.method.annotation.MutationMapping;
import org.springframework.stereotype.Controller;

@Controller
@RequiredArgsConstructor
public class UserController {

    private final UserService userService;

    @MutationMapping
    AuthResponse login(@Argument LoginInput input) {
        return userService.login(input);
    }
}
```

Restart and sign in as the seeded administrator:

```graphql
mutation {
  login(input: { username: "admin", password: "admin123" }) {
    token
    user { username role }
  }
}
```

The response contains a long token string. This is how every protected call will authenticate from now on, so the workflow is worth practicing once:

1. Copy the `token` value from the response (without the quotes).
2. In GraphiQL, open the **Headers** tab at the bottom of the query editor, next to Variables.
3. Enter the header as JSON, with the word `Bearer`, a space, and the token:

```json
{
  "Authorization": "Bearer eyJhbGciOiJIUzI1NiJ9..."
}
```

Every request GraphiQL sends now carries this header, the `JwtAuthFilter` validates it, and the security context knows you are `admin` with `ROLE_ADMIN`. The header persists until you clear the tab or the token expires (24 hours in our configuration). The frontend picks the mutation up as well: a Sign in page appears, complete with the demo accounts.

## 6. Authorization: protecting the mutations

Authentication established who is calling; authorization decides what they may do. Our rule: everyone may read, only administrators may modify the catalog. This is expressed with one annotation per protected method, and we place it on the **service**, not the controller:

`src/main/java/com/graphqlguy/moviedb/movie/MovieService.java` (and the same on `createPerson`, `updatePerson`, and `delete` in `PersonService`)

```java
import org.springframework.security.access.prepost.PreAuthorize;

@Transactional
@PreAuthorize("hasRole('ADMIN')")
Movie createMovie(final CreateMovieInput input) {
```

Why the service layer? Because a rule attached to the business method holds **regardless of which caller reaches it**. The GraphQL resolver is one entry point today; a scheduled job, a REST controller, or a second resolver may appear tomorrow, and none of them can bypass a check that lives on the service itself. The controller remains what it has always been: a thin layer that translates GraphQL into method calls. (This is also why `@EnableMethodSecurity` from step 4 matters: it is what makes these annotations execute at all.)

`hasRole('ADMIN')` matches the `ROLE_ADMIN` authority that `JwtAuthFilter` placed in the security context; the `ROLE_` prefix is added by the `hasRole` check automatically.

Verify all three outcomes in GraphiQL with the same mutation:

```graphql
mutation {
  createMovie(input: { title: "Test", releaseYear: 2020, genre: DRAMA }) {
    id
    title
  }
}
```

- With the Headers tab empty, the mutation is rejected with the classification `UNAUTHORIZED`. Note that this error is clean and structured, unlike the `INTERNAL_ERROR` we saw in class 3: Spring for GraphQL ships a dedicated resolver for security exceptions.
- Log in as `user` (password `user123`), put that token in the Headers tab, and run the mutation again: rejected with `FORBIDDEN`. Authenticated, but not authorized.
- Replace the header with the `admin` token: the mutation succeeds.

Queries need no header at all, exactly as intended. In the frontend, the admin features (Add Movie, Edit, Delete) now appear only after signing in as an administrator.

## Recap

- Spring Security blocks everything by default; a `SecurityFilterChain` bean opens deliberately: CSRF off and sessions stateless because JWT carries the identity, all URLs permitted because authorization is enforced per method.
- `JwtAuthFilter` turns a valid `Authorization: Bearer` header into a populated security context, and leaves anonymous requests to proceed; rejection is authorization's job.
- `@EnableMethodSecurity` activates `@PreAuthorize`; without it the annotations are silently ignored.
- `login` issues signed tokens; `@PreAuthorize("hasRole('ADMIN')")` on the service methods guards writes regardless of the entry point.
- Anonymous callers receive `UNAUTHORIZED`, authenticated non-admins receive `FORBIDDEN`, and reads remain open to everyone.
